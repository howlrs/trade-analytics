### Maker Rebates (負のメイカー手数料)

メイカー出来高の割合に応じたリベート:

| Maker Volume比率 | Rebate |
|-----------------|--------|
| >0.5% | -0.001% (1bps) |
| >1.5% | -0.002% (2bps) |
| >3.0% | -0.003% (3bps) |

**結論: Yes、Hyperliquidはメイカーリベートを提供する。** 出来高条件を満たせば注文を出すだけで報酬を得られる。

### ステーキング割引

HYPEトークンをステーキングすることで追加の手数料割引:
- Wood (>10 HYPE): 5%割引
- Bronze/Silver/Gold/Platinum: 段階的に増加
- Diamond (>500,000 HYPE): 40%割引

### 特殊ペア割引

- **安定通貨ペア**: Taker手数料/Maker Rebate/出来高寄与が80%低減
- **Aligned quote assets**: Taker 20%低減、Maker Rebate 50%改善、出来高寄与20%増

## 2. API

### REST API
- エンドポイント: `https://api.hyperliquid.xyz`
- 情報取得: POST `/info` (マーケットデータ、アカウント情報)
- 取引: POST `/exchange` (注文発注・キャンセル)

### WebSocket
- エンドポイント: `wss://api.hyperliquid.xyz/ws`
- リアルタイムオーダーブック、約定、ポジション更新

### Python SDK
- **公式SDK**: `pip install hyperliquid` (PyPI)
- GitHub: https://github.com/hyperliquid-dex/hyperliquid-python-sdk
- 同期/非同期両方をサポート
- ccxtにもHyperliquid統合あり

### 低レイテンシ最適化
- 自前のノードを立てて、ノード出力から直接オーダーブック等を構築
- APIコールよりも高速かつ高粒度
- nonce invalidation (noop) でキャンセルを最適化

## 5. レイテンシ

### サーバーロケーション
- **東京 (日本)** がプライマリノードロケーション
- 低レイテンシには東京にノードを設置することが推奨

### パフォーマンス
- **ブロックタイム**: 0.07秒 (70ms)
- **エンドツーエンドレイテンシ** (co-located):
  - 中央値: 0.2秒 (200ms)
  - 99パーセンタイル: 0.9秒 (900ms)
- **スループット**: ~200,000 注文/秒

### 最適化手段
- 東京にco-locatedノードを設置
- 自前ノードからオーダーブックを構築 (API経由より高速)
- 最低スペック: 32論理コア、500MB/sディスクスループット
- RPC提供者 (Chainstack, Dwellir等) が東京/シンガポールにエッジサーバー提供

**注意**: 200msの中央値レイテンシは、CEX (Binance ~1ms) と比べると2桁遅い。CEXクオリティのHFT market makingは困難。

## 6. 注文タイプ

| 注文タイプ | 説明 |
|-----------|------|
| Market | 即時約定 |
| Limit (GTC) | Good Till Canceled |
| **Post-Only (ALO)** | Add Liquidity Only。即時マッチする場合はキャンセル |
| **IOC** | Immediate or Cancel。未約定部分はキャンセル |
| Stop Market | トリガー価格でマーケット注文 |
| Stop Limit | トリガー価格でリミット注文 |
| Scaled Order | 価格範囲にわたり複数注文を分散配置 |
| TWAP | 30秒間隔のサブ注文に分割 |

### レバレッジ制限
- Market注文: ~10x - 25x (ノーショナルサイズ依存)
- Limit注文: 最大10x

### MM向け重要ポイント
- **Post-Only (ALO)がある** - メイカーリベートを確実に得るために必須
- キャンセルとPost-Only注文はGTC/IOCより**優先的に処理される** (L1レベルで強制)

## 7. 決済 (Settlement)

- **USDC建て** - 全ポジションはUSDCで担保
- 全てのマッチングと決済はオンチェーンでアトミックに実行
- **ブリッジ**: Arbitrum経由でUSDCを入出金
- ガスレス取引 (ガス代不要)

## 8. スマートコントラクトリスク

### アーキテクチャ
- **独自L1ブロックチェーン** (HyperBFT consensus, HotStuff派生)
- **HyperCore**: オーダーブック・マッチングエンジン
- **HyperEVM**: EVM互換レイヤー

### 監査状況
- **公式ドキュメントに監査情報の記載なし**
- Ethereum等の確立されたL1と比較してテスト・精査が不十分
- 独自L1のため、Ethereumのセキュリティモデルに依存しない

### TVL
- ピーク: $3.4B (2024年12月)
- 現在: $1.5B-$2B程度 (2026年3月)

### リスク要因
- Arbitrumブリッジコントラクトの脆弱性
- L1自体のバグ・ダウンタイム
- オラクル操作リスク (バリデーターがオラクルを維持)
- コンセンサス障害時のダウンタイム

### 緩和策
- OIキャップ (流動性・ベーシス・レバレッジに基づく)
- オラクル価格から1%以上離れた注文は配置不可
- バリデーター21ノード (DPoS)

## 9. MEV/フロントランニングリスク

### 主要な保護メカニズム

**注文優先順位の強制 (L1レベル)**:
- キャンセルとPost-Only注文がGTC/IOCより**先に処理される**
- これはL1自体が強制するルール。ノードが正しくブロックを実行するには、この順序を守る必要がある
- メイカーはキャンセルが高確率で成功することを確信できる

**結果**:
- HFTメイカー vs HFTテイカーの取引率が他プラットフォームの**1/10以下**
- 従来型のMEV (サンドイッチ攻撃等) は構造的に困難

### 残存リスク
- バリデーターがブロック提案者としてトランザクション順序を操作する理論的可能性
- ただしHyperBFTのリーダーローテーション (200msごと) が緩和
- オンチェーンオーダーブックなので、全注文が可視 (情報非対称性なし)

## Market Making適性の総合評価

### メリット
1. **メイカーリベートあり** (-0.001% ~ -0.003%)
2. **Post-Only注文サポート**
3. **キャンセル優先処理** (stale quoteリスク軽減)
4. **高出来高** ($3.7B+/日)
5. **BTC/ETH/SOL全対応**
6. **ガスレス取引**
7. **東京サーバー** (日本からの接続に有利)

### デメリット
1. **レイテンシがCEXの100倍以上** (200ms vs 1-2ms)
2. **スマートコントラクト/L1リスク** (監査情報不明)
3. **ブリッジリスク** (Arbitrum経由)
4. **Limit注文レバレッジ最大10x**
5. **TVL減少傾向** ($3.4B -> $1.5B)
6. **全注文がオンチェーンで可視** (戦略が見える)

### 推奨
- CEXのHFT market makingとは異なるアプローチが必要
- より広いスプレッドでの passive MM が適切
- レイテンシ競争ではなく、スプレッド・在庫管理で差別化
- 東京にノードを設置してレイテンシ最適化推奨
